# ERP 系统一周任务分配表

> 人员：5 人  
> 时间：约 7 天  
> 目标：完成整体原型的约 50%～60%，保证核心页面能演示  
> 开发方式：每个人独立负责一个纵向模块的数据库、后端接口和前端页面

> 更细的表结构、接口、页面、每日任务和验收项见 [DETAILED_TASK_ASSIGNMENT.md](DETAILED_TASK_ASSIGNMENT.md)。

## 1. 一周内要完成的范围

### 必做功能

- 登录、退出和基础角色菜单。
- 首页框架、侧边菜单和基础统计卡片。
- 商品、客户、供应商、仓库基础资料。
- 销售单及销售单详情。
- 采购单及采购单详情。
- 入库、出库、库存查询和库存流水。
- 收款单、付款单和简单资金流水。
- 销售、采购、库存三个基础报表。
- 操作日志或个人信息页面至少完成一个。

### 直接放弃或只做静态页面

- 注册和复杂企业资料完善流程。
- 销售退货、采购退货的完整业务闭环。
- 借入、借出。
- 复杂盘点、调拨和库存调整审批。
- 客户及供应商完整对账逻辑。
- 消息通知。
- 复杂角色数据权限。
- 高级报表、利润精确核算和 Excel 导入导出。
- 反审核、红冲、打印和通用工作流。

## 2. 五人任务分配

| 成员 2 | 基础资料 | 商品、分类、单位、客户、供应商、仓库的表和 CRUD 接口 | 商品、客户、供应商、仓库列表及新增/编辑弹窗 | 至少完成商品、客户、供应商、仓库四组可用 CRUD |
| 成员 3 | 销售和收款 | 销售单、销售明细、收款单接口；销售单状态修改；简单销售汇总查询 | 销售单列表、新增、详情；收款单页面；销售报表页面 | 能创建销售单、查看详情、修改状态和登记收款 |

成员编号可以直接替换成真实姓名。

## 3. 公共约定

为避免五个人生成出五套不同风格的代码，第一天必须统一以下内容：

| 项目 | 统一约定 |
| --- | --- |
| 后端包名 | 按 `system`、`masterdata`、`sales`、`purchase`、`inventory`、`finance` 分包 |
| 接口前缀 | `/api/v1` |
| 返回格式 | `code`、`message`、`data` |
| 分页参数 | `page`、`size` |
| 时间格式 | `yyyy-MM-dd HH:mm:ss` |
| 金额类型 | 数据库 `DECIMAL(18,2)`，Java `BigDecimal` |
| 数量类型 | 数据库 `DECIMAL(18,4)`，Java `BigDecimal` |
| 前端组件 | Vue 3、Element Plus、Pinia、Axios |
| 数据库修改 | 每个人提交自己模块的 SQL，表名必须带模块前缀 |
| Git 分支 | 每人使用自己的 `feature/模块名` 分支，不直接修改 `main` |

## 4. 七天安排

| 时间 | 全组目标 | 每个人要做的事 |
| --- | --- | --- |
| 第 1 天上午 | 公共骨架可运行 | 成员 1 创建前后端工程；全组确定接口、目录、页面风格和数据库命名 |
| 第 1 天下午 | 五个模块同时启动 | 每人建立自己模块的数据表、后端目录和前端路由，先提交空页面和接口框架 |
| 第 2 天 | 基础 CRUD | 完成各自核心表的新增、查询、修改、删除，并让前端列表显示真实数据 |
| 第 3 天 | 核心业务页面 | 完成新增单据、单据详情、状态修改或库存增减等主要流程 |
| 第 4 天 | 模块基本完成 | 补表单校验、筛选、分页、错误提示；提交可独立演示的模块 |
| 第 5 天 | 第一次全量合并 | 合并到集成分支；统一菜单；处理接口冲突、跨模块字段和数据库脚本 |
| 第 6 天 | 联调与演示数据 | 修复阻断问题；补充演示账号、商品、客户、供应商、采购、销售和库存数据 |
| 第 7 天 | 最终交付 | 部署、走完整演示流程、录屏或截图、整理 README、答辩 PPT 和人员贡献说明 |

## 5. 每日协作规则

- 每天至少合并一次代码，不要等到第 5 天才第一次提交。
- 每天开始前用 10 分钟说明：昨天完成了什么、今天做什么、有什么阻塞。
- 每天结束前由每个人演示自己的模块，页面能打开但不能操作不算完成。
- 公共文件由成员 1 维护；其他人修改路由、菜单、全局配置前先同步。
- 跨模块需要的数据先使用固定 ID 或简单下拉接口，不等待复杂联动。
- 遇到复杂功能先降级：保留页面和基本 CRUD，暂时删除审批、回滚和高级统计。
- 第 5 天以后原则上不增加新功能，只修复会影响演示的问题。

## 6. 建议的演示主线

最终答辩时按一条完整路径演示，避免逐个点击所有页面：

1. 使用管理员账号登录。
2. 新增一个商品、客户、供应商和仓库。
3. 创建采购单并执行入库，展示库存增加和库存流水。
4. 创建销售单并执行出库，展示库存减少和库存流水。
5. 登记一笔客户收款和供应商付款。
6. 打开首页、销售报表、采购报表和库存报表查看结果。
7. 展示用户管理或操作日志作为系统管理功能。

只要这条演示主线稳定，项目就已经具备较完整的进销存系统效果。

## 7. 完成度控制

### 目标完成度约 60%

| 范围 | 预计完成情况 |
| --- | --- |
| 登录和后台框架 | 完成 |
| 基础资料 | 完成主要资料 |
| 销售 | 完成销售单，退货简化或不做 |
| 采购 | 完成采购单，退货简化或不做 |
| 库存 | 完成出入库、余额、流水；其他功能选做 |
| 资金 | 完成基础收付款；复杂对账不做 |
| 报表 | 每类完成一个基础表格或图表 |
| 设置 | 完成用户角色和一个辅助页面 |

### 功能降级顺序

时间不够时按以下顺序删除：

1. 图表动画和页面美化。
2. 盘点、调拨、退货、其他收支。
3. 高级筛选和批量操作。
4. 精细按钮权限和数据权限。
5. 非核心报表。

登录、商品、采购、销售、库存余额和库存流水不要删除，因为它们组成最基本的演示闭环。
