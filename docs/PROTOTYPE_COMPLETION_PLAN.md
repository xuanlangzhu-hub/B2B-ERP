# ERP 原型完整业务实施计划

## 1. 目标

在现有可运行主项目上继续开发，将思维导图中的页面和业务尽量全部落地。开发完成后，再按照五名成员的职责拆分可运行副本和讲解资料。

本轮不再接受“只有 Entity、Mapper、空 Controller”作为完成。一个功能只有同时具备以下内容才算完成：

1. 数据库表和演示数据可用；
2. 后端接口、参数校验、状态校验和事务完整；
3. 前端列表、查询、新增、详情和业务按钮可操作；
4. 菜单、路由和权限编码一致；
5. 至少包含核心服务测试或可重复的接口验证；
6. 前后端构建通过，能够使用演示账号走通流程。

## 2. 当前基线

### 已形成闭环

- 登录、JWT、用户、角色、菜单、操作日志；
- 商品、客户、供应商、仓库基础维护；
- 销售单：创建、修改、审核、出库、库存扣减、生成应收；
- 采购单：创建、修改、审核、入库、库存增加、生成应付；
- 库存余额、库存流水、入库单、出库单；
- 账户、收款、付款、自动核销、资金流水和冲销；
- 销售退货、采购退货及对应库存和往来处理；
- 销售、采购、库存基础统计页面。

### 基线验证

- 后端 `mvn test` 通过；
- 前端 `npm run build` 通过；
- 当前数据库结构已经预留盘点、调拨、调整、借用等表，无需重新设计整个数据库。

## 3. 固定开发顺序

每个业务模块都按照下面的顺序完成：

1. 核对表结构、状态值、唯一约束和索引；
2. 编写实体、DTO、VO、Mapper；
3. 编写 Service，并把库存、金额和状态更新放在同一事务；
4. 编写 Controller，统一使用 `/api/v1`；
5. 补前端 API 类型和请求函数；
6. 补列表、查询、表单、详情及状态按钮；
7. 增加路由、菜单和权限编码；
8. 增加演示数据和测试；
9. 完成后端测试、前端构建和人工业务回归；
10. 一个闭环完成后单独提交，方便后期拆分成员副本。

## 4. 阶段一：库存盘点

这是下一步立即实施的功能。

**状态：已完成。** 后端测试、前端生产构建以及本地 MySQL 接口生命周期验证均已通过。

### 后端

- 盘点单分页、详情、新建、修改、删除；
- 从指定仓库自动生成账面库存明细；
- 保存实际数量，自动计算盘盈、盘亏；
- 提交盘点、审核盘点、取消盘点；
- 审核后更新库存余额；
- 生成 `COUNT_GAIN`、`COUNT_LOSS` 库存流水；
- 同一仓库存在进行中盘点时禁止重复创建；
- 已审核盘点禁止重复记账。

### 前端

- 盘点单列表和状态筛选；
- 新建盘点并选择仓库；
- 录入实际数量，实时展示差异数量；
- 盘点详情、提交、审核、取消；
- 审核结果可以在库存余额和库存流水中验证。

### 验收闭环

`库存为 10 → 新建盘点 → 实盘 8 → 审核 → 库存变为 8 → 产生数量 -2 的盘亏流水`。

## 5. 阶段二：库存调拨与商品报损

**状态：已完成。** 后端测试、前端类型检查与生产构建均已通过；本地 MySQL 已完成创建、审核/取消/删除接口生命周期验证。

### 库存调拨

- 调拨单列表、详情、新增和取消；
- 来源仓和目标仓不能相同；
- 审核时校验来源仓库存；
- 同一事务扣减来源仓、增加目标仓；
- 同时生成 `TRANSFER_OUT` 和 `TRANSFER_IN` 两条流水；
- 前端展示调拨前后仓库、商品、数量和状态。

### 商品报损/库存调整

- 调整单列表、详情、新增、审核和取消；
- 支持报损、盘盈修正、盘亏修正和其他调整；
- 减库存时禁止形成负库存；
- 审核后写库存流水并记录原因；
- 前端按原型提供调整类型、仓库、商品、数量和备注。

### 验收闭环

- A 仓 10 件调拨 3 件到 B 仓后，A 仓为 7、B 仓增加 3，并存在成对流水；
- 报损 2 件后库存减少 2，重复审核不会重复扣减。

实现说明：自动化服务测试已覆盖调拨完成时的双仓库存与成对流水、库存不足时的整体回滚，以及报损减少、增加新库存和库存不足场景。接口联调使用临时草稿单完成创建、审核/取消和删除验证，测试数据已清理。

## 6. 阶段三：借入、借出和归还

**状态：已完成。** 借出与借入使用独立导航入口，共用完整业务页；后端测试、前端生产构建和本地 MySQL 借出—归还全流程验证均已通过。

### 业务范围

- 借入单、借出单分页和详情；
- 创建、审核、出入库、归还、取消；
- 记录往来单位、预计归还日期、已归还数量和未归还数量；
- 借出减少库存，借出归还增加库存；
- 借入增加库存，借入归还减少库存；
- 禁止超量归还；
- 逾期和未还清状态可查询；
- 对应库存流水能够追溯来源单据。

### 验收闭环

完成一笔借出及部分归还，再完成一笔借入及全部归还，库存数量和单据剩余数量一致。

实现说明：审核借出会扣减库存，审核借入会增加库存；归还支持按商品分批办理，自动累计已归还和未归还数量并生成正反向库存流水。自动化测试覆盖借出扣库、借入新建库存、部分归还及超量归还拦截；真实接口验证确认完整归还后库存数量和成本金额恢复，临时数据已清理。

## 11. 阶段八：最终验收与成员副本

### 全链路验收

至少完整演示以下流程：

1. 采购下单 → 审核 → 入库 → 应付 → 付款；
2. 销售下单 → 审核 → 出库 → 应收 → 收款；
3. 销售退货 → 退货入库 → 冲减应收；
4. 采购退货 → 退货出库 → 冲减应付；
5. 盘点 → 盘盈盘亏 → 库存流水；
6. 调拨 → 双仓库存变化；
7. 借出/借入 → 归还；
8. 报表数字与业务单据、库存和资金一致。

### 质量门槛

- 后端测试通过；
- 前端类型检查和生产构建通过；

- 配置文件不包含真实服务器密码或私钥；
- README 包含本地启动、账号、数据库初始化和演示顺序；
- 核心状态操作具备幂等和并发保护；
- 所有写操作具备企业数据隔离。
